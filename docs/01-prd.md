## 数据流

### 配件搜索流程（三种入口）

```
方式 1: 车架号搜索
输入 VIN → 后端解析车型（品牌/车型/年份/发动机）
       → 匹配 product_vehicle_fits 表
       → 返回适配配件列表（按类别分组）

方式 2: 文字搜索
输入 OEM编号 / 配件名 → 全文搜索 products 表
                    → 匹配 name_zh / name_en / oem_number
                    → 返回结果列表

方式 3: 车型浏览
选择品牌 → 选择车型 → 选择年份 → 选择类别树节点
        → 逐级筛选 product_vehicle_fits → products
        → 返回结果列表
```

### 供应商上架流程
```
注册 → 上传资质(营业执照+税号) → 自动OCR提取税号
→ 与官方数据交叉验证 → 自动 verified → 创建产品(绑定车型适配) → 上架
```

### 采购商下单流程（询问制）

```
采购商浏览产品 → 查看详情（含供应商信息、保修/退换政策）
→ 提交询问订单（提供 OEM 号、货品图片对比、车型信息等）
→ 供应商接入询问，核对细节，确认能否供货
→ 双方确认后生成正式订单
→ 采购商进入支付界面
→ 采购商通过 PayPal / Apple Pay / 支付宝海外付等方式支付
→ 平台根据支付方式接收付款邮件 → OCR 抓取支付信息
→ 与采购商上传的支付截图比对
   ├── 比对成功 → 订单状态 = paid
   └── 比对不成功 → 标记待人工审核
→ 供应商确认收款 → 发货(关联货运+运单号)
→ 货运追踪更新 → 收货确认
```

### 支付状态模型

```
未支付 (unpaid)
  └── 采购商进入支付界面（但未实际支付）
        → 订单保持 unpaid → 定时通知采购商 → 48h 后自动提醒供应商

已支付待验证 (payment_submitted)
  └── 采购商上传支付截图 + 平台收到支付邮件
        → OCR 抓取截图与邮件数据比对
           ├── 金额/时间/收款方匹配 → verified (已支付)
           ├── 部分匹配 → pending_review (待人工审核)
           └── 未找到对应支付记录 → 继续等待邮件或人工介入

已支付 (paid)
  └── 供应商确认收款 → 进入发货流程
```

### 供应商拒绝流程

```
供应商收到询问订单 → 接入沟通细节 → 确认能否供货
→ 若无法供货（缺货/型号不符/价格变动）→ 拒绝并附原因
→ 采购商收到通知 → 平台推荐同类产品
```

### 纠纷 / 退款流程

```
采购商收货后不满意 → 查看产品页供应商写明的保修/退换政策
→ 采购商与供应商协商
   ├── 协商一致 → 退货/换货/退款（线下执行）
   └── 协商失败 → 任一方向平台申请客服介入
        → 平台客服查看订单记录、双方凭证
        → 做出调解建议 → 双方执行
```

### 图片上传流程
```
前端上传 → POST /api/products/:id/images → 后端生成唯一文件名
→ 上传 Cloudflare R2 → 保存 URL 到 product_images → 返回 URL
```

## 未纳入 v1.0

| 功能 | 原因 |
|------|------|
| 平台托管支付（资金池） | v1 做邮件+截图比对验证，不碰资金 |
| 多语言（除中/英） | v1 先做中英双语 |
| 供应商评级系统 | 需要交易数据积累 |
| 移动端 App | 先验证 Web 端 |
| 实时聊天 (WebSocket) | 站内消息够用 |

| 功能 | 理由 |
