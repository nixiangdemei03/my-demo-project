# APEX — PRD v1.0

> Auto Parts EXport · 中国汽配供应商 → 海外采购商 B2B 平台

---

## 1. 目标

**解决什么问题**：国内中小汽配厂商有出口资质但缺海外渠道；海外汽配批发商想找高性价比中国货源但现有平台（阿里巴巴国际站）门槛高、搜索噪音大、运费不透明。

**给谁解决**：

| 角色 | 画像 | 语言 | 核心需求 |
|------|------|:--:|----------|
| 🏭 供应商 | 国内汽配厂/贸易商销售经理 | 中文优先 | 快速上架产品、管理订单、对接货运 |
| 🌍 采购商 | 海外汽配进口商/批发商 | 英文优先 | 浏览产品、VIN/OEM 精准搜索、下单追踪物流 |
| 🔧 管理员 | 平台运营 | 中/英 | 审核供应商、管理内容、查看数据 |

**衡量指标（v1.0 上线后首个完整季度）**：

| 指标 | Q1 目标 |
|------|:--:|
| 注册供应商 | ≥ 20 |
| 上架产品 | ≥ 200 |
| 注册采购商（来自 ≥3 国家） | ≥ 50 |
| 完成订单 | ≥ 10 |
| 页面 LCP | < 3s |

**商业模式**：平台统一收银 → 梯度佣金（前 40 供应商 3%，后续 7%）→ 每月 1 日结算 → 半年度风险评估。

---

## 2. 非目标

以下明确 **不做**（防止 scope creep）：

| 不做 | 原因 |
|------|------|
| 多语言（除中/英） | v1 先做中英双语 |
| 供应商评级系统 | 需要交易数据积累 |
| 移动端 App | 先验证 Web 端 |
| 实时聊天 (WebSocket) | 站内消息够用 |
| 批量导入产品 (Excel/CSV) | 供应商 SKU 量大时必需，但不阻塞上线 |
| 货运公司 API 自动追踪 | 初期手动更新追踪事件即可 |
| 天眼查 OCR 自动验证 | 非免费公开 API，改用人工审核 |
| 房源/社交/支付托管 | 专注 B2B 汽配交易 |

## 4. 验收标准

### 4.1 核心数据流

**配件搜索（三种入口）**：

```
方式 1 — 车架号搜索 (VIN)
采购商输入 VIN → 平台用 vin_pattern 前缀匹配 product_vehicle_fits 表
  ├── 命中 → 返回适配配件列表（按类别分组）
  └── 未命中 → 提示"暂无该 VIN 的适配数据"
              → 采购商仍可下单，订单中附带 VIN
              → 供应商在订单中看到 VIN，自行用解码工具查询车型

方式 2 — 文字搜索 (OEM/关键词)
输入 OEM 编号 / 配件名 → 全文搜索 products 表
  → 匹配 name_zh / name_en / oem_number → 返回结果列表

方式 3 — 车型浏览（采购商输入 + 供应商纠正闭环）
采购商输入已知信息（品牌/车型/年份）→ 浏览匹配配件
  → 下单时附带 VIN → 供应商对比货品图片与 VIN → 纠正/补全车型信息
  → 交易完成或咨询结束后 → 弹窗邀请供应商填写该 VIN 的车型适配数据
  → 系统录入 → 下次同车型自动匹配
  → 填写数据的供应商获得推流奖励（公式后续定义）
```

**供应商上架流程**：

```
邮箱注册 → 收邮件验证链接 → 激活账号 → 上传资质文件（营业执照、实体店照片、销售许可证扫描件/照片）
  → 进入人工审核队列 → 管理员审核证照真实性与一致性
  → 审核通过 → verified = true → 创建产品(绑定车型适配) → 上架
  → 审核不通过 → 附原因退回，供应商可重新提交
```

> 邮箱验证通过 MailChannels + Cloudflare Workers 免费发送，邮件中包含 10 分钟有效的一次性验证链接。

> v1.0 不接入天眼查 OCR 自动验证。人工审核是主路径，后期可接入工商 API 做预填辅助。

**采购商下单流程（询问制）**：

```
采购商浏览产品 → 查看详情（含供应商信息、保修/退换政策）
  → 提交询问订单（提供 OEM 号、货品图片对比、车型信息等）
  → 供应商接入询问，核对细节，确认能否供货
  → 双方确认后生成正式订单
  → 采购商进入支付界面 → 平台统一收银台支付（Stripe/PayPal/支付宝国际版占位 API）
  → 支付成功 → webhook 回调 → 订单状态自动变为 paid
  → 供应商确认收款到账 → 发货(关联货运+运单号)
  → 货运追踪更新 → 收货确认
```

**询问转单流程**：

```
采购商提交询问（含 OEM、图片、车型信息等）→ 供应商接入沟通细节
  → 供应商确认能否供货：
      ├── 能供货 → 采购商在询问单界面点击「下单」→ 自动转为正式订单（带入询问中的信息）
      └── 无法供货 → 采购商或供应商点击「取消」
           → 询问记录在双方端均保留，状态显示为"已取消"
           → 采购商收到通知 → 平台推荐同类产品
```

**支付状态模型**：

```
采购商下单 → 订单生成，状态为 pending，payment_status = unpaid
  └── 前端显示倒计时 timer：24h 内未完成支付 → 自动取消
  └── 供应商端订单列表明确标注「未付款」字样，供应商可自行联系采购商确认意向

支付流程：
  采购商点击支付 → 调用支付网关占位 API
  → 支付成功 → webhook 回调更新 payment_status = paid → 订单状态 → confirmed
  → 支付失败 → 返回失败原因，订单保持 unpaid

已支付 (paid)
  └── 款项进入平台账户 → 自动确认 → 供应商发货 → 收货后进入结算周期

注：v1.0 支付网关为模拟占位 API，真实对接在后续版本完成。
```

**多币种支持**：

```
供应商以 CNY（人民币）上传原价
  └── 系统根据中国银行实时汇率转换为 USD 售价展示
  └── 采购商可在支付时选择支付币种（USD / EUR / GBP 等）
  └── 结算统一以 USD 为基准货币
```

**平台结算模型**：

```
每月 1 日结算上月已收货订单
  └── 日流水记录：每日根据实际成交情况生成日流水（供应商/采购商/平台三方各持一份），用于对账
  └── 月总销售额 = Σ 所有已收货订单金额
  └── 月度结算 = 汇总当月每日流水

佣金梯度（基于供应商注册顺序）：
  └── 前 40 个注册供应商：平台抽成 = 月总销售额 × 3%（早期激励）
  └── 第 41 个起：平台抽成 = 月总销售额 × 7%（标准费率）
  └── 供应商 ID 用于判定注册顺序

供应商实际到账 = 月总销售额 - 平台抽成
  └── 平台出具结算单 → 供应商确认 → 批量打款

半年度供应商风险评估：
  └── 每半年根据流水数据评估供应商经营状况
      ├── 销售额过低 → 平台主动沟通，提供推流帮助
      └── 销售额异常过高 → 检查采购商与供应商双方，防止洗钱或资产转移

注：汽配行业供应商均提前备货（头年备次年库存），海外发货链路为
厂商 → 国际物流 → 海外，不经过供应商仓库。月度结算不挤压供应商现金流。
```

**抽成示例（前 40 供应商，3% 费率）**：

```
某供应商月销售额 $50,000，月中发生 1 笔 $3,000 退款
退款流程：采购商退货 → 供应商决定是否需要检测报告
         ├── 不需要 → 供应商直接对接厂家退货 → 确认收到退货 → 进入退款
         └── 需要 → 供应商收货检测 → 提供检测报告/定损 → 确认定损结果
         → 无争议 → 从供应商当月结算中扣除 $3,000
月结算基数 = $50,000 - $3,000 = $47,000
  → 平台抽成 = $47,000 × 3% = $1,410
  → 供应商实际收到 = $47,000 - $1,410 = $45,590
```

**纠纷 / 退款流程**：

```
采购商收货后不满意 → 查看产品页供应商写明的保修/退换政策
  → 采购商与供应商协商退货
     ├── 供应商不需要检测报告 → 直接对接厂家退货 → 确认收到退货 → 进入退款
     └── 供应商需要检测报告 → 收货检测 → 出具报告/定损 → 确认后进入退款
  → 退款金额从供应商当月结算中扣除
  → 若协商失败 → 任一方向平台申请客服介入
        → 平台客服查看订单记录、双方凭证 → 做出调解建议 → 双方执行

数据模型：orders 表含 refund_status 字段（none / requested / approved / rejected / completed）
API：退款相关端点预留占位（/api/refunds），v1.0 为模拟 API
独立 Issue：Issue 14（Refund & Return Flow）
```

**图片上传流程**：

```
前端上传 → POST /api/products/:id/images → 后端生成唯一文件名
  → 上传 Cloudflare R2 → 保存 URL 到 product_images → 返回 URL
```

### 4.3 技术栈

> 全栈基于 Cloudflare 免费套餐，零服务器成本上线。

| 层 | 技术 | 理由 |
|----|------|------|
| Backend | **Hono** (TypeScript) | FastAPI 风格路由，原生支持 Cloudflare Workers |
| Database | **Cloudflare D1** | 免费 5GB / 500 万读行/天 / 100 万写行/天，SQLite 兼容 |
| ORM | **Drizzle ORM** | TypeScript-first，原生支持 D1，类 SQL 语法 |
| 校验 | **Zod** | 端到端类型安全，与 Hono 深度集成 |
| Auth | JWT (**jose** + **bcryptjs**) → httpOnly Cookie | Workers Web Crypto API，Set-Cookie 写入（JS 不可读），Access 15min / Refresh 7d，浏览器自动携带 |
| 邮件 | **MailChannels** + Workers | 免费发送邮箱验证邮件 |
| Migration | **Drizzle Kit** | Drizzle 自带迁移工具 |
| Frontend | React 18 + Vite 5 | 快速 HMR，构建速度快 |
| Storage | Cloudflare R2 | S3 兼容、零出站费 |
| Deploy | Cloudflare Pages + Workers | 全球 CDN、免费额度覆盖 MVP |

> **与旧版差异**：FastAPI (Python) → Hono (TypeScript)，PostgreSQL → D1 (SQLite)，SQLAlchemy → Drizzle ORM，Pydantic → Zod，python-jose → jose。变更原因为 Cloudflare Workers 不支持 Python 运行时，且 D1 是 Cloudflare 生态唯一的免费数据库。

### 4.4 安全设计

| 区域 | 措施 |
|------|------|
| 密码 | bcryptjs 哈希，最小 8 位，必须含字母+数字 |
| JWT | jose 签名 (HS256)，Workers Web Crypto API，Access Token 15min / Refresh Token 7d |
| Token 传输 | httpOnly Cookie（JS 不可读），浏览器自动携带，前端无需管理 token |
| CSRF | `SameSite=Lax` + 修改操作校验 `X-CSRF-Token` |
| 邮箱验证 | 注册必须验证邮箱，MailChannels 发送 10 分钟有效一次性链接 |
| API 鉴权 | Hono 中间件读 Cookie 中 access_token，失败返回 401 |
| 数据所有权 | 供应商只能编辑自己的产品；采购商看自己的订单 |
| 图片上传 | 类型白名单 (jpg/png/webp)，最大 5MB |
| CORS | 同域部署 `/api/*`，无需额外 CORS 配置 |
| XSS | React 默认转义 + CSP Header |

## 5. 风险与未知

| # | 风险 | 影响 | 当前状态 | 应对 |
|:--:|------|------|:--:|------|
| 1 | **支付网关商务对接** — Stripe/PayPal/支付宝国际版的跨境合规要求、资金分账方案、持牌机构接入 | 高 | 🔴 未启动 | Phase 1 并行推进商务沟通；v1.0 代码层用占位 API |
| 2 | **供应商 UUID 无法排序** — "前 40 个"依赖自增序列号判定，UUID 没有自然顺序 | 中 | 🟡 已知 | users 表加 `registration_seq` 自增字段 |
| 3 | **后台定时任务无调度** — 月度结算生成、24h 订单超时取消依赖后台任务机制 | 中 | 🟡 已知 | Phase 2 前确定方案（Cloudflare Workers Cron / APScheduler） |
| 4 | **通知系统缺失** — SUP-08 提醒、支付催付都依赖统一推送层 | 中 | 🟡 延后 | Phase 3 设计通知表 + 推送通道 |
| 5 | **冷启动** — 没有供应商就没有产品，没有产品就没有采购商 | 高 | 🟡 需行动 | 开发团队自行寻找种子用户；前 40 供应商 3% 低佣金激励 |
| 6 | **VIN 数据库冷启动** — 初期 vin_pattern 覆盖率低，大部分 VIN 命中不了 | 中 | 🟢 有方案 | 交易后弹窗邀请供应商填写 + 推流奖励（公式 TBD） |
| 7 | **多币种汇率波动** — 实时汇率变化可能导致采购商看到的 USD 价格和实际结算价不一致 | 低 | 🟢 有方案 | 建议每日固定汇率（中国银行牌价），避免实时波动纠纷 |
| 8 | **Payment webhook 伪造** — 接入真实支付网关时必须验证 webhook 签名 | 中 | 🟡 已知 | 届时按 Stripe/PayPal 文档加签名验证 |
| 9 | **推流奖励公式 TBD** — VIN 数据贡献和低销推流的具体算法未定义 | 低 | 🟢 延后 | Phase 2 末期定义 |
| 10 | **D1 SQLite 兼容性** — 无 JSONB/DECIMAL/ENUM 类型，原 PostgreSQL 数据模型需适配 | 中 | 🟡 已知 | TEXT 存 JSON（D1 有 JSON 函数），INTEGER 存金额（分），VARCHAR + 应用层 CHECK 替代 ENUM |
| 11 | **Workers 冷启动延迟** — 免费版 Worker 平均冷启动 ~50ms | 低 | 🟢 可接受 | B2B 非高频场景影响小，API 响应延迟仍在可接受范围 |
| 12 | **MailChannels 免费额度变更** | 低 | 🟢 有备选 | 备选 Resend（100 封/天）或 Brevo（300 封/天），均为免费层 |

## 6. 最小版本 (MVP) 定义

> **原则**：先做"能跑通的骨架"，再做"完整的闭环"。MVP 分两阶段交付——第一阶段上线产品目录（供应商能上架、采购商能搜），第二阶段上线交易链路（能下单、能发货、能追踪）。

### 6.1 MVP-1：产品目录（Phase 1，Week 1-2）

**一句话**：供应商能注册、上架产品；采购商能注册、搜索浏览产品。

| 维度 | 内容 |
|------|------|
| **Issue** | #1 Scaffolding · #2 DB Models · #3 User Auth · #4 Product CRUD · #5 Search · #8 Supplier Page · #9 Frontend Catalog · #10 Supplier Dashboard |
| **涉及用户故事** | SUP-01/02/03/06/09/10 · BUY-01/02/03/09/10/11 · ADM-01 |

**核心用户路径**：

```
🏭 供应商：
  注册 → 上传资质 → 管理员审核 → 创建产品(含图片/车型适配) → 上架 → 管理库存

🌍 采购商：
  注册 → 浏览首页 → VIN/OEM/文字/车型浏览 搜索 → 查看产品详情 → 查看供应商主页

🔧 管理员：
  审核供应商资质(通过/退回)
```

**MVP-1 交付物清单**：

| 模块 | 具体功能 | 状态 |
|------|---------|:--:|
| 脚手架 | FastAPI + React + Vite + `make dev` 联调 | P0 |
| 数据库 | 5 核心表 + Alembic migration | P0 |
| 用户系统 | Cookie-based JWT 注册/登录/刷新/登出 + bcrypt + 邮箱验证 + CSRF | P0 |
| 产品 CRUD | 创建/编辑/删除/列表 + R2 图片上传 + 车型适配绑定 | P0 |
| 搜索 | VIN 前缀匹配 + OEM/文字全文搜索 + 类别树车型浏览 | P0 |
| 供应商主页 | 模板化主页(Logo/介绍/资质/联系方式) + 审核 API | P0 |
| 前端目录 | 首页(搜索入口) · 搜索结果页 · 产品详情页 · 供应商页 · 登录/注册页 | P0 |
| 供应商后台 | 产品管理(列表/新增/编辑) · 订单管理(占位) · 主页编辑 | P0 |

**MVP-1 明确不做**：

| 不做 | 原因 |
|------|------|
| 订单创建/状态流转 | 属于 MVP-2 |
| 货运公司/发货/追踪 | 属于 MVP-2 |
| 采购商下单/支付 | 属于 MVP-2 |
| 询问订单/站内消息 | 属于 MVP-2 |
| 管理后台统计/类别管理 | 属于 v1.0 完整版 |
| 通知系统 | 属于 v1.0 完整版 |

### 6.2 MVP-2：交易闭环（Phase 2，Week 3-4）

**一句话**：在 MVP-1 基础上，采购商能下单、支付、追踪物流；供应商能接单、发货。

| 维度 | 内容 |
|------|------|
| **Issue** | #6 Order API · #7 Freight · #11 Buyer Dashboard · #13 Payment & Settlement · #14 Refund Flow · #15 Inquiry→Order |
| **涉及用户故事** | SUP-04/05/07 · BUY-04/05/06/07/12 |

**核心用户路径**：

```
🌍 采购商：
  浏览产品 → 提交询问订单(含 OEM/图片/车型) → 供应商确认 → 转正式订单
  → 支付(占位 API) → 查看订单状态 → 追踪货运

🏭 供应商：
  查看收到的订单 → 确认/拒绝 → 收到支付确认 → 选择货运公司 + 填运单号 → 发货
  → 查看物流追踪状态
```

**MVP-2 交付物清单**：

| 模块 | 具体功能 | 状态 |
|------|---------|:--:|
| 订单管理 | 创建/查询列表/状态流转(待确认→已确认→已发货→已收货) + 24h 超时自动取消 | P0 |
| 货运对接 | 货运公司 CRUD + 发货关联运单号 + 追踪事件更新与展示 | P0 |
| 采购商后台 | 订单列表/详情 + 支付收银台(占位 API) + 货运追踪 + 询问历史 | P0 |
| 支付结算 | 统一收银台占位 API + webhook 回调更新状态 + 月度结算逻辑 | P0 |
| 询问转单 | 结构化询问表单(含 OEM/图片/车型) → 供应商确认 → 一键转正式订单 | P0 |
| 退款流程 | 退款申请/供应商处理/结算扣减(占位 API) | P0 |

**MVP-2 明确不做**：

| 不做 | 原因 |
|------|------|
| 真实支付网关对接 | 商务推进中，先用占位 API |
| 货运公司 API 自动追踪 | 初期手动更新追踪事件 |
| 管理后台(除审核外) | 属于 v1.0 完整版 |
| 通知推送 | 属于 v1.0 完整版 |

### 6.5 MVP 度量标准

| 里程碑 | 硬性标准 |
|--------|---------|
