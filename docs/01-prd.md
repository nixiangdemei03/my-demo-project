## 数据流

### 配件搜索流程（三种入口）

```
方式 1: 车架号搜索
采购商输入 VIN → 平台用 vin_pattern 前缀匹配 product_vehicle_fits 表
            ├── 命中 → 返回适配配件列表（按类别分组）
            └── 未命中 → 提示"暂无该 VIN 的适配数据"
                         → 采购商仍可下单，订单中附带 VIN
                         → 供应商在订单中看到 VIN，自行用解码工具查询车型

方式 2: 文字搜索
输入 OEM编号 / 配件名 → 全文搜索 products 表
                    → 匹配 name_zh / name_en / oem_number
                    → 返回结果列表

方式 3: 车型浏览（采购商输入 + 供应商纠正闭环）
采购商输入已知信息（品牌/车型/年份）→ 浏览匹配配件
→ 下单时附带 VIN 给供应商 → 供应商对比货品图片与 VIN → 纠正/补全车型信息
→ 订单交易完成后或咨询结束后 → 弹窗邀请供应商填写该 VIN 对应的车型适配数据
→ 系统录入纠正后的车型适配数据 → 下次同车型自动匹配
→ 填写了数据的供应商获得推流奖励（奖励公式后续定义）
```

### 供应商上架流程
```
注册 → 上传资质文件（营业执照、实体店照片、销售许可证等扫描件/照片）
→ 进入人工审核队列 → 管理员审核证照真实性与一致性
→ 审核通过 → verified = true → 创建产品(绑定车型适配) → 上架
→ 审核不通过 → 附原因退回，供应商可重新提交
```
> 注：v1.0 不接入天眼查 OCR 自动验证。人工审核是主路径，后期可接入工商 API 做预填辅助。

### 采购商下单流程（询问制）

```
采购商浏览产品 → 查看详情（含供应商信息、保修/退换政策）
→ 提交询问订单（提供 OEM 号、货品图片对比、车型信息等）
→ 供应商接入询问，核对细节，确认能否供货
→ 双方确认后生成正式订单
→ 采购商进入支付界面
→ 采购商通过平台统一收银台支付（参考支付宝海外付 / PayPal / Stripe）
→ 支付网关对接持牌机构做资金合规分账
→ 款项进入平台统一账户 → 订单状态自动变为 paid
→ 供应商确认收款到账 → 发货(关联货运+运单号)
→ 货运追踪更新 → 收货确认
```

### 支付状态模型

```
采购商下单 → 订单生成，状态为 pending，payment_status = unpaid
  └── 前端显示倒计时 timer：24h 内未完成支付 → 自动取消
  └── 供应商端订单列表展示此订单，明确标注「未付款」字样，
      供应商可自行联系采购商确认意向

支付流程：
  采购商点击支付 → 调用支付网关（Stripe/PayPal/支付宝国际版占位 API）
  → 支付成功 → webhook 回调更新 payment_status = paid → 订单状态 → confirmed
  → 支付失败 → 返回失败原因，订单保持 unpaid

已支付 (paid)
  └── 款项进入平台账户 → 自动确认
        → 供应商发货 → 收货后进入结算周期

注：v1.0 支付网关为模拟占位 API，真实对接在后续版本完成。
支付 webhook 端点已在 API 设计中预留。
```

### 多币种支持

```
供应商以 CNY（人民币）上传原价
  └── 系统根据中国银行实时汇率转换为 USD 售价展示
  └── 采购商可在支付时选择支付币种（USD / EUR / GBP 等）
  └── 结算统一以 USD 为基准货币
```

### 平台结算模型

```
每月 1 日结算上月已收货订单
  └── 日流水记录：每日根据实际成交情况生成日流水（供应商/采购商/平台三方各持一份），用于对账
  └── 月总销售额 = Σ 所有已收货订单金额
  └── 月度结算 = 汇总当月每日流水，生成月度流水

佣金梯度（基于供应商注册顺序）：
  └── 前 40 个注册供应商：平台抽成 = 月总销售额 × 3%（早期激励）
  └── 第 41 个起：平台抽成 = 月总销售额 × 7%（标准费率）
  └── 供应商 ID（UUID 主键）的数字部分用于判定注册顺序

供应商实际到账 = 月总销售额 - 平台抽成
  └── 平台出具结算单 → 供应商确认 → 批量打款

半年度供应商风险评估：
  └── 每半年根据流水数据评估供应商经营状况
      ├── 销售额过低 → 平台主动沟通原因，提供推流帮助
      └── 销售额异常过高 → 检查采购商与供应商双方，防止洗钱或资产转移风险

注：汽配行业供应商均提前备货（头年备次年库存），海外发货链路为
厂商 → 国际物流 → 海外，不经过供应商仓库。月度结算不挤压供应商现金流。
```

**抽成示例（前 40 供应商，3% 费率）**：
```
某供应商月销售额 $50,000，月中发生 1 笔 $3,000 退款
退款流程：采购商退货 → 供应商决定是否需要检测报告
         ├── 不需要 → 供应商直接对接厂家退货 → 确认收到退货 → 进入退款
         └── 需要 → 供应商收货检测 → 提供检测报告/定损 → 确认定损结果
         → 无争议 → 从供应商当月结算中扣除 $3,000
月结算基数 = $50,000 - $3,000 = $47,000
  → 平台抽成 = $47,000 × 3% = $1,410
  → 供应商实际收到 = $47,000 - $1,410 = $45,590
```

### 询问转单流程

```
采购商提交询问（含 OEM、图片、车型信息等）→ 供应商接入沟通细节
→ 供应商确认能否供货：
    ├── 能供货 → 采购商在询问单界面点击「下单」→ 自动转为正式订单（带入询问中的信息）
    └── 无法供货 → 采购商或供应商点击「取消」
         → 询问记录在双方端均保留，状态显示为"已取消"
         → 采购商收到通知 → 平台推荐同类产品
```

### 纠纷 / 退款流程

```
采购商收货后不满意 → 查看产品页供应商写明的保修/退换政策
→ 采购商与供应商协商退货
   ├── 供应商不需要检测报告 → 直接对接厂家退货 → 确认收到退货 → 进入退款
   └── 供应商需要检测报告 → 收货检测 → 出具报告/定损 → 确认后进入退款
→ 退款金额从供应商当月结算中扣除
→ 若协商失败 → 任一方向平台申请客服介入
        → 平台客服查看订单记录、双方凭证
        → 做出调解建议 → 双方执行

数据模型：orders 表含 refund_status 字段（none / requested / approved / rejected / completed）
API：退款相关端点预留占位（/api/refunds），v1.0 为模拟 API
独立 Issue：退款扣减验收标准已从 Issue 13 中拆分（见 Issue 14）
```
