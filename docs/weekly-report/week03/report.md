## 2. 本周学到了什么（知识 + 反思）

- **PRD 不是写出来的，是问出来的**：8 轮 brainstorm 每轮只聚焦一个维度（支付→退款→VIN→供应链），逐个消除模糊。R5 从"平台不碰钱"翻盘到"统一收银"是整个迭代最重要的转折——不碰钱就没有 2% 抽成，没有抽成就没有商业模式。
- **AI 复审 PRD 能发现人忽略的系统性问题**：我自己自评 8.8 分，Claude 交叉比对 6 份文档后给了 6.5 分——支付/询问/结算三块核心缺失是我完全没意识到的。不是因为写得差，是因为人脑很难同时追踪 PRD↔数据模型↔API 设计↔Issues 四层一致性。
- **我最 aha 的瞬间**：担心"月度结算会挤压供应商现金流"，Claude 指出汽配行业供应商提前一年备货，海外发货链路是厂商→国际物流→海外，不经过供应商仓库。我之前基于 B2C 电商经验的假设在这个行业完全不存在。
- **我之前以为原型做完就完了**，这周发现原型必须和 PRD 校准——PRD 改了 14 处，原型跟着改了 9 处。不校准的话原型就是"上一版 PRD 的图"，写代码时会产生两种理解。
- **我还没完全搞懂的**：支付网关商务对接不是技术问题——找谁聊、聊什么、协议模板目前全部空白。这是 Phase 1 期间需要并行启动的非技术卡点。

## 3. AI 协作记录

- **最爽的一次**：把完整 PRD + 5 份配套文档丢给 Claude 做交叉一致性审查，它 30 秒内就发现支付系统在 PRD 里写了但 API 和数据模型完全没设计——这是我自己看了两天没看出来的。
- **最坑的一次**：R2 时 Claude 在 PRD 里写了"平台自动解析 VIN 解码车型"，技术文档写得漂亮但实际不存在免费好用的全球 VIN API。R4 我发现供应商自己就有 VIN 解码工具，Claude 才改为 vin_pattern 前缀匹配 + 透传兜底。教训：**AI 会默认互联网上存在它想象出来的 API**，涉及外部数据源的方案必须亲自验证。
- **个人套路升级**：脑暴→评估→修改→再评估→确认→写入 PRD → **交叉复审**。第二轮 PRD 修改完成后让 Claude 再做了一轮"PRD ↔ 数据模型 ↔ API ↔ Issues ↔ 原型"的五向对齐，发现 9 处原型不一致。这个校验环让我对"改了文档之后别忘了改关联文件"有了肌肉记忆。

## 4. 决策记录

| 决策 | 选了 | 放弃了 | 理由 |
|------|------|--------|------|
| 支付模型 | 平台统一收银 | OCR 截图验真 | 统一收银才能抽成；OCR 验真 30% 失败率 |
| VIN 解码 | vin_pattern + 供应商兜底 | 平台自动解码 | 无免费全球 VIN API |
| 佣金模型 | 前 40 供应商 3%，后续 7% | 统一 2% | 梯度激励冷启动，后期商业化 |
| 供应商审核 | 人工审核（执照+实店照片+许可证） | 天眼查 OCR 自动验证 | 天眼查 API 非免费公开，OCR 准确率不稳定 |
| 车型数据库 | 人工输入 + 交易后纠正闭环 | 购买 TecDoc | 年费 €5000，v1 用不起 |
| 退款决策 | 供应商自主判断是否需要检测 | 强制检测 | 供应商自己最清楚需不需要检测 |
| 消息系统 | 轻量 message + 重量 inquiry 两层 | 统一消息模型 | 自由咨询和正式询盘的字段需求不同 |
