# /check-prd

## 这个 skill 做什么

读取一份 PRD 文件，从三个维度审查并输出结构化报告：

1. **板块完整性** — 是否包含 5 个必须板块（目标/非目标/用户故事/验收标准/风险）
2. **验收标准可测性** — 每条 AC 是否可被自动或人工明确验证
3. **配套文档一致性** — 如果存在配套文档（data-model / api-design / roadmap），交叉比对矛盾点

## 输入

- 一份 PRD 文件路径（如 `docs/01-prd.md`）
- 可选：配套文档路径（如 `docs/03-data-model.md` `docs/04-api-design.md`）

## 审查维度详解

### 板块完整性

| 板块 | 检查点 |
|------|------|
| 目标 | 有没有写"解决什么问题"？有没有写目标用户？有没有可量化指标？ |
| 非目标 | 有没有明确列出 ≥3 项不做的事？有没有防 scope creep 的边界线？ |
| 用户故事 | 是否 "作为 X，我想 Y，以便 Z" 格式？是否有优先级 P0/P1/P2？三个角色是否都覆盖？ |
| 验收标准 | 是否有 ≥1 条可验证的 AC？是否关联到用户故事？ |
| 风险 | 是否列出已知会卡的点？是否有不确定的待定项？ |

### 验收标准可测性

| ❌ 模糊词（不可测） | ✅ 可测替代 |
|---------|---------|
| "正常" "好用" "快速" "流畅" | 状态码 / 响应时间 / 文件路径 / URL |
| "支持多种格式" | "jpg/png/webp → 201；gif → 400" |
| "用户能登录" | "POST /api/auth/login 正确密码 → 200 + token" |
| "性能好" | "LCP < 3s" / "API 响应 < 500ms" |
| "大部分用户满意" | "反馈分 ≥ 4.0/5.0" |

### 配套文档一致性

交叉比对 PRD 与技术文档，检查：
- PRD 提到的功能是否有对应的 API 端点 / 数据表？
- PRD 的结算模型数字是否与 API 设计中的计算逻辑一致？
- PRD 的优先级 P0 是否在 Roadmap 中分配了 Phase？
- 数据模型字段是否覆盖 PRD 用户故事所需的数据？

## 必须遵守

- 每条问题必须引用 PRD 原文（不要自己复述）
- 验收标准问题必须给出**具体改写建议**
- 信息不足时不编造——标注"需用户确认"
- 配套文档一致性只检查已存在的文件，不要求用户提供不存在的东西
- 报告输出到 `docs/superpowers/specs/YYYY-MM-DD-prd-review.md`

## 反模式

- 输出"PRD 整体不错"这种无信息量的结论 → 必须逐条引用原文
- 只说"缺验收标准"不说缺哪条的 → 标注具体用户故事 ID
- 改写 AC 时脱离 PRD 原意自己发挥 → 紧扣原文意图
- 把主观偏好当问题（"我觉得应该用表格"）→ 只报客观缺陷
